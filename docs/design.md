## 1. プロジェクトの性格(最重要・全判断の前提)

- **個人の実験場**。OSS 運営は前提にしない。注目されなくてよい
- README・doc comment は**日本語のみ**で構わない
  - 理由: 想定読者は作者一人。将来読者が現れるなら日本語で Lean を学ぶ人であり、
    その場合も日本語が正しい
- 命名は気負わない普通の英単語。凝った名前・マイナーな単語は採用しない
- リポジトリ名 `lean-shed`(検索の埋もれ回避)、名前空間 `Shed.*`
- 誰も依存していないうちはリネーム無料。育ったら考え直す
- **エージェントへの注意**: このプロジェクトを「改善」と称して壮大化させないこと。
  設計議論で最も繰り返された修正が「現実の規模への引き戻し」だった

## 2. 設計原則

1. **契約の正本は Lean の型** — 外部とのやり取りは structure +
   deriving ToJson/FromJson で定義し、境界で FromJson 再検証。
   配管側(Python 等)は契約に従う使い捨て可能な部品
2. **表層は具体的、深層は一般的** — 入口 API はモノモーフィック
   (String / Array / FilePath)。型クラス抽象は下層に隠す
3. **Pure / Sys 二階建て** — `Shed.Pure.*`(Lean のみ、証明可能)と
   `Shed.Sys.*`(外界接触、信頼ノード)を import 構造で分離。
   Pure のみ import するコードは検証可能性が構造的に保たれる
4. **昇格ルール** — 実プロジェクトで**二回**書いたものだけ収録。推測で書かない
5. **予測可能性 > 網羅性** — 一つ覚えれば全部に通用する規約を優先。
   一貫した規約は人間にも AI(文脈内学習)にも効く

## 3. 横断規約

- **三段活用**: `parse`(Except)/ `parse?`(Option)/ `parse!`(panic)を全
  データ形式系で統一。`Module.動詞` で補完から当てられる命名
- **エラー統一**: Sys 層は共通の `Shed.Error` 階層。失敗モードを doc で列挙
- **全公開関数に doc comment + 実行可能 example**(`#guard` で CI 検証)
- **ブラケット必須**: 資源(ハンドル・ソケット・子プロセス)は `withXxx` で
  確保と解放を対にする
- **デフォルト有界**: タイムアウト・リトライ・サイズは既定で有限、無限はオプトイン。
  停止性証明の思想の IO への延長。AI 製コードの暴走は「間違える」より
  「正しいことを無限にやる」形で現れがち、への対策

## 4. 外界との接続方針

- **第一選択: サブプロセス**(JSON over stdin/stdout)
  - 単発: stdin に書いて閉じ(EOF)、stdout を読み切る。exit != 0 はエラー
  - 高頻度: 常駐ワーカー + 行区切り JSON。stderr は .inherit
  - LLM API のようにネットワーク I/O が支配的な用途では
    プロセス起動オーバーヘッドは誤差
- **FFI(@[extern])は限定使用**: ホットループ(毎秒数万回級)か
  巨大データのゼロコピー共有のみ。サイレントなメモリ破壊が証明済みコードと
  同居するリスクを許容する場面に限る
- **Haskell / Go ランタイム同居は封印**(GHC RTS 管理 / cgo 境界の苦行に見合わない)

## 5. 検証の勾配(コントロールの壁)

正しさの保証は一枚岩ではなく、コストの異なる層の積み重ね:

    型 < エフェクト < 有界性 < 事後条件 < 証明

- **型**: 最安・全自動。**事前条件はここに吸収される**(依存型・証明引数として
  引数型に載せ、コンパイル時に呼び出し側へ強制)。純粋関数なら
  サブタイプ `{x : α // P x}` で事後条件もここに載りうる
- **エフェクト**: 「何に触りうるか」の上限宣言。仕様の安価な近似(§6)
- **有界性**: 「どれだけ使うか」(§3)
- **事後条件**: IO 操作が外界に約束する結果。IO では戻り値の型に外界への影響を
  載せる標準手段がないため、定理・doc・テストとして別建てになる。
  勾配でここだけ独立しているのはそのため
- **証明**: 最高価。重要度が見合う箇所のみ

**不変表明が独立項目でない理由**: Lean では不変条件はデータ型設計に溶かすのが常道
(「ソート済みリスト」は不変条件つきリストではなく SortedList 型にする)。
ループ不変条件相当も再帰の補題として現れる。Hoare 三点セットのうち
事前・不変は型とデータ設計に吸収され、IO の事後条件だけが「別途書く仕様」として残る。

**運用**: shed の Sys 層の最低ラインは「エフェクト + 有界性 + 失敗モードの doc 列挙」。
完全な事後条件・証明は要求しない(実験場の規模に見合わない)。
AI が実装を書く比率が上がるほど、勾配の下層(機械検査可能な部分)が
人間のレビューを肩代わりする、という位置づけ。

## 6. エフェクト(オプトイン)

- 言語強制は不可能(@[extern] / unsafe の穴)→ 規約として運用
- 方式: ケイパビリティ型クラスの二枚看板
  - 直球: `Shed.Http.get : String → IO String`
  - 規律版: `Shed.Cap.Http` 配下に `[MonadHttp m] → String → m String`
- 「Sys 層の外で生 IO を使わない」を lint で縛る(プロジェクトごと有効化)
- mathlib 文化圏(IO を書かない)でエフェクトが盛り上がらないのは構造的事情であり、
  実用コードでの価値とは別問題

## 7. モジュール計画

- `Shed.Pure.Text.Unicode` — 正規化(NFC/NFKC)、文字種判定。
  `Text.Ja`(ひらがな/カタカナ/漢字判定、簡易文分割)を含む。

## 9. Lean 4 技術知見(調査済み・再調査不要)

### モナド変換子
- ReaderT / StateT / ExceptT / OptionT は標準。Id がベース(StateM σ = StateT σ Id)
- **WriterT は標準になく mathlib 側**。Lean は正格なのでログ蓄積は
  StateT 代替が定石(→ §11 の WriterT 不使用の理由)
- MonadReaderOf / MonadStateOf は型で層を区別、同種変換子の二枚重ねが可能
- 持ち上げは MonadLift の自動挿入(mtl の n² インスタンスと異なる方式)
- StateRefT: ST 風可変参照の高速版。コンパイラ自身が多用

### サブプロセス
- stdin ハンドルは参照カウントで最終使用後に自動クローズ → EOF。
  知らないとワーカーが入力待ちで固まる。Python 側は flush=True 必須
- 常駐ワーカーの stderr を .piped にするとバッファ詰まりでデッドロックし得る
  → .inherit にしてある

### その他
- CJK 識別子は «» で囲む
- エラボレーション時に任意の IO を実行可能 → 外部仕様(OpenAPI 等)から
  コンパイル時に型生成するマクロが原理的に書ける(将来の拡張候補)
- 型による検索(exact? / Loogle)は自分のライブラリ+依存に今日効く。
  shed が育つほど「名前でなく型で引く」が使い心地になる

## 10. 将来の接続点: 日本語 Lean 記事

- 第一弾の構想: **「はず」の形式化**。中心主張は
  「『A のはずだ』は無前提の証明可能性(⊢ A)ではなく、話者の背景知識からの
  相対的導出(Γ ⊢ A)であり、Γ が誤りうるため外れうる」という分析。
  直観主義論理を知る読者が最初に踏む罠をそのまま導入にできる
- 根拠の例: 直接証拠があるとき「はず」は使えない(かばんを見た直後に
  「入れたはずだ」とは言わない)/「来るはずだったのに来なかった」が成立する
- **土台文献の候補**(書誌は記憶ベース、執筆前に実物確認が必要):
  - 記述言語学: 益岡隆志『モダリティの文法』、仁田義雄のモダリティ研究、
    日本語記述文法研究会の文法書(モダリティ巻)
  - 形式側: Artemov の正当化論理(justification logic、t:A 形式は
    「根拠つき主張」として「はず」分析と直結)、
    McCready & Ogata の日本語証拠性表現の形式意味論、
    Simpson の直観主義様相論理(学位論文、定番)
  - 構成: 記述言語学の分析 → 正当化論理/直観主義様相論理への翻訳 → Lean 実装
- `Text.Ja` はこのプロジェクトの実需。記事 → 小さなライブラリという
  非同期・低社交コストの貢献経路が前提で、shed の運営方針(§1)と整合

## 11. やらないことリスト(理由つき)

- 線形型もどき・セッション型などの重い機構 — 表層 API を確実に壊す
- WriterT — 標準にない+正格言語ではサンク/タプル蓄積が不利。StateT で代替
- 英語ドキュメント — 想定読者が日本語話者(§1)
- 推測ベースのモジュール先行実装 — 二回ルール違反。AI なら埋められるが、
  埋まること自体に戦略的価値はない(価値は規約と勾配の設計にある)
- 注目獲得のための活動 — 要件に入っていない。設計原則は注目と逆方向に最適化済み

## 13. 追記(2026-07): 契約カーネルへの再定位

立ち上げ直後の対話で判明した事実により、優先順位を組み替えた。
§1〜12 は歴史的記録として原文のまま残す(リポジトリ名 `lean-shed` 表記等は当時のもの。実名は `lean4-shed`)。

### 変わった前提
- 作者(データエンジニア)は Python をほぼ手書きしなくなった。仕事の大半が
  AI を中間に置いて行われる
- そこで生じている実害: 理解負債、文書の混乱とやり直し、責任を取るべき場面
  での根拠の不在。「日本語のまま試行を繰り返す」ことへの限界感
- 直近の実プロジェクトは dlt / dbt / Dagster によるデータ連携改善

### 診断
問題は AI がコードを書くことではなく、**出力の合否判定が人間の注意力と
自然言語の曖昧さに課金され続ける**こと。型検査・証明検査は合否を注意力から
独立させる固定点であり、これは §5 の一文
「AI が実装を書く比率が上がるほど、勾配の下層が人間のレビューを肩代わりする」
の実地適用である。ピボットではなく、設計の核心への合流。

### 決定
1. **第一級トラック = 契約カーネル**(`Shed.Pure.Contract`)。
   データ契約を Lean の型で正本化し、dbt schema tests / JSON Schema 等を生成。
   正本が小さく・一つで・機械検査されるなら、実装がどれだけ churn しても
   理解負債は契約層で止まる
2. **エンジンは移植せず運転する**。polars 相当は pure Lean で書かない。
   DuckDB(SQL)を主経路、Python ワーカー(`Sys.Py`)を脱出ハッチとし、
   「Python に落ちても制御フローと型は Lean から出ない」に目標を緩和する
3. **二回ルールの改訂**: 実装の執筆コストは AI によりほぼ消滅したため、閾値の
   対象を「執筆労力」から「API 表面の一貫性とレビュー注意力」に変更。
   職業標準セット(データエンジニアの日常 import 相当)は事前承認とし、
   それ以外は従来どおり実需駆動
4. **消費者なきモジュール禁止**: 全モジュールは examples/ か tests/ に
   実行可能な消費者を持つ。エージェントの投機的実装への構造的な歯止め
5. 「はず」形式化・正当化論理のトラックは休眠可・削除不可。
   AI が証明を書く時代に証明を読める能力が責任能力の源泉になる、
   という位置づけに変わった(演繹の筋トレ)

### 教訓(§12 と同型)
仕様は発言ではなく成果物と行動から掘る。「顧客が望むものを顧客は知らない」は
作者自身にも、設計を書いた AI にも適用される。判定は常に縦切り一枚の
実装を実需に接触させて行う。削除は無料。

## 14. 追記(2026-07): 正本の向きの反転(作者の指摘による)

§13 の契約カーネルは「Lean が正本 → dbt を生成」だったが、作者から
「dbt(SQL)を正本とし、manifest JSON から Lean をマクロ生成して、
dbt tests にできない検証を Lean でやる」という逆向きの提案があり、
**実プロジェクトの文脈ではこちらが正しい**と判断した。dbt は既にチームの
正本であり、Lean は既存パイプラインに変更を求めない下流の検証・理解レイヤー
として導入できる。

- 実装: `Shed.Pure.Dbt`(manifest の型・レイヤー規約)+
  `Shed.Sys.Dbt`(`def_dbt_project` / `dbt_check` — エラボレーション時 IO で
  manifest を取り込む。§9 の「外部仕様からコンパイル時に型生成」の実地)
- `dbt_check` はレイヤー規約違反を**コンパイルエラー**にする。
  lake build そのものが契約ゲートになる
- 両向きは共存する。Lean が駆動する個人ツールでは生成方向(§13)、
  既存 dbt プロジェクトでは取り込み方向(本節)。使われない方は削除する
- 境界規約を追加: 「道具は知ってよい、プロジェクトは知らない」。
  命名規約等は `Conventions` で注入し、ハードコードしない

なお、作者の関心リストのうち「一般の SQL 等価性判定」は研究フロンティア
(決定不能・Cosette 系)であり道具トラックには置かない。実務は
データ差分で代替し、関係代数の小さな断片の意味論は筋トレトラック行き。

## 15. 追記(2026-07): 初回接触の還流(社内 dbt プロジェクト)

検証器を社内に持ち込む形で初めて実プロジェクトに接触し、
機密を含まない「道具への苦情」が3点還流された
(持ち込み手順書は役目を終えたため削除済み):

1. `Conventions.stagingPrefix : String` では多層命名を表現できない
2. `depType` が private でカスタム規則が書けない
3. `Layer` の staging/mart 二値は粗いが、述語ベースなら実用上足りる

対応: `Conventions` を述語注入型(`isStaging : Node → Bool`)へ改修、
複数接頭辞は `Conventions.ofPrefixes` で表現。`Project.depType` と
`NodeType.isRaw` を公開。`Layer` 型は削除(苦情 3 の通り不要だった)。

教訓: 接頭辞 `Array String` 化ではなく述語化を選んだ。「接頭辞で分類できる」
という仮定自体が一度実プロジェクトに負けており、同じ仮定に二度賭けない。
このループ(持ち込み → 苦情だけ還流 → 即日改修)が §13-14 の設計の
運用形態として機能することが確認できた。

## 16. 追記(2026-07): 受容宣言(2回目の接触の還流)

2回目の持ち込み検証(独自の多層命名を持つ実プロジェクト)の還流:
述語注入は機能した(depType 再実装も不要になった)が、検出された違反の
一部(mart 層によるマスタ source 直読み等)は意図的な設計判断であり、
例外を表明できない限り「コンパイルが通れば規約違反ゼロ」に到達できない、
という指摘を受けた。

対応: `Waiver`(model / dep / **reason 必須**)と `runRulesWith`、
`dbt_check ... accepting "waivers.json"` を追加。設計判断:

1. **理由必須** — 例外は「見逃し」ではなく署名済みの判断として残す。
   中心目的(責任の所在の明示)の直接の実装
2. **未使用の宣言はエラー** — 違反が解消されたのに宣言が残るのは
   受容リストの腐敗。lint の抑制コメントや許可リストが放置されて
   実態と乖離していく問題への構造的対策
3. これにより緑(検査が通った状態)の意味が強くなる:
   「未処理の違反ゼロ + 全例外が署名済み + 例外リストが実態と一致」

`Rule` の返り値は `Violation`(modelId / depId / message)に構造化した。
宣言との照合を文字列マッチにしないため。

## 17. 追記(2026-07): 実プロジェクト初の緑(運用フェーズ入り)

3回目の接触で、実プロジェクト(千モデル規模)の検証が
初めて緑に到達した。受容宣言 14 件適用、腐敗防止(未使用宣言の検出)も
実機で動作確認済み。dbt 検証トラックは開発フェーズを終え、運用フェーズに入る:

- waivers.json / Check.lean の正本は**消費者リポジトリ側**に置く
  (「道具は知ってよい、プロジェクトは知らない」の運用形)
- 推定で書かれた reason はチームレビュー(waivers.json の PR)で確定させる。
  暗黙の設計判断の明文化こそこのトラックの本来の成果物
- 消費者側 CI に Check.lean を置けば、以後の検査は人手の往復なしで回る

### バックログ(実害が出るまで作らない)

- `dbt_check ... accepting` は既定規約固定。規約注入(`Conventions`)との
  併用はランタイムの `runRulesWith` に落ちる。コンパイル時ゲートで規約注入
  したくなったら、elab 時の項評価(`evalExpr Conventions`)で拡張する。
  CI 運用では exit code で同等のゲートになるため優先度は低い

## 18. 追記(2026-07): 確定需要の原語は一度で完成させる(作者の指摘による)

「glob や正規表現のように使うことが確定しているものは、中途半端にせず
完成させてほしい」という作者の指摘を受け、8 割ルールを精密化した。
8 割ルールの本来の標的は「仕様が開いていて完成を定義できないもの」であり、
確定需要の原語で 8 割止めをすることは節約ではなく摩擦の先送り
(欠けた機能に出会うのは必ず別の作業の最中)である。

規約(CLAUDE.md に追加): **確定需要の原語は、閉じた仕様を宣言して
一度で完成させる**。

- **glob** — 完成済み。仕様 = fnmatch 相当(`*` `?` `[...]` `[!...]` + `**`)。
  Python の fnmatch をオラクルにした突き合わせテストつき。
  ブレース展開 `{a,b}` はシェル拡張であり glob 仕様外
- **正規表現(次スライス、仕様確定済み)** — `Shed.Pure.Regex`。
  仕様 = **正規言語の範囲**: リテラル・`.`・文字クラス・エスケープ
  (`\d` `\w` `\s` 等)・アンカー(`^` `$`)・量指定子(`* + ? {m,n}`、
  貪欲/怠惰)・選択・キャプチャ/非キャプチャ群。
  **後方参照と先読みは永久に対象外**(正規言語でなく、線形時間照合が
  原理的に不可能)。実装は PikeVM(Thompson NFA + キャプチャ追跡)で
  最悪でも線形時間 — bounded-by-default の照合器。
  表層 API: `compile`(Except)/ `test` / `find?` / `findAll` / `replace` / `split`。
  検証: #guard + Python `re` をオラクルにした突き合わせ(対象は共通部分集合)
- **CLI 引数(プールに追加)** — 閉じた仕様(フラグ・位置引数・ヘルプ生成)を
  切れば同規約の対象。実装順は正規表現の後

### §18 の改訂: 正規表現の「最良の完成」はエンジンの運転(作者の指摘による)

作者から「PCRE を常用するので最良の方法で作り切ってほしい」との指摘。
PikeVM 案(正規言語の部分集合)では先読み・後読み・後方参照という
常用機能が永久に手に入らないため、方針を変更した:

- **`Shed.Sys.Regex` を実装(完成)** — Python `re` を常駐ワーカーとして
  運転。PCRE 級の全機能が初日から使え、エンジン = オラクルなので意味論の
  乖離が原理的に起きない。「エンジンは移植せず運転する」の適用形
- `Pure.Regex`(PikeVM、線形時間)は**純粋文脈の実需**(規則述語での
  パターン照合、信頼できない入力)が立った時に併設する。役割が違うので
  Sys.Regex とは競合しない

### §18 の補足: FFI への移行条件(計測済み)

「ここまでやるなら FFI では」という作者の問いに対する実測と判断:

- ワーカー方式の実測: 往復型(1 件ずつ test)で秒間約 1.6 万回、
  バッチ型(findAll 一発)で秒間約 10 万件抽出
- データ仕事の大半はバッチ型であり、現設計で足りる。往復型が
  §4 の FFI 基準(毎秒数万回級)に実測で達したら、**表層 API を
  変えずに** `Re` の裏側を PCRE2 FFI に差し替える(表層は単相で
  固定済みのため可逆)
- 中間手当(必要時): バッチ API(`testMany` 等)の追加、
  巨大ファイルの行フィルタは `grep -P` の運転

## 19. 追記(2026-07): CI 非対応の例のカテゴリ(手元検証専用)

`examples/SheetToAnki.lean`(Google Sheets を正本にした Anki デッキの
片方向 upsert)を取り込んだ。これで examples/ には二種類の消費者が混ざる:

- **CI で実行できる消費者**(LogPipeline / Contracts / DbtChecks)—
  自前で立てたローカルサーバーや合成フィクスチャで往復まで検証する
- **CI で実行できない消費者**(SheetToAnki)— 実 Anki の起動・AnkiConnect
  アドオン・Google Sheets という外部の生きた依存を必要とし、CI では
  `--run` できない

後者の扱い(規約):

- モジュールヘッダに「CI 非対応(手元検証専用)」と実行コマンドを明記する
- `ci.yml` では `--run` の実行リストからは外すが、`lake env lean <file>`
  (実行なしの型検査)は回す。examples/ は `lake build` の対象外
  (`lean_lib Shed` のみ)なので、型検査を CI に載せないと消費者が
  黙って腐る。`#guard` もこの型検査で走る
- 「消費者なきモジュール禁止」の趣旨は満たす — 実行はできなくても、
  本体 API(この例では `Data.sqlIdent` / `sqlStr`、`Http.postJson`、
  `Data.queryAs`)の一貫性を型で押さえる消費者として機能する

副産物: SheetToAnki が `Data.sqlIdent` / `sqlStr` の実需第 1 号になり、
`insertRows` にインラインで埋まっていたクォート処理を公開関数へ切り出した
(§3「確定需要の原語」ではなく、消費者が生まれてからの昇格。CLAUDE.md
「昇格則」通り)。`queryAs` の NULL 落とし穴も doc に明記した(空セルは
JSON null になり String フィールドで落ちる → coalesce か Option String)。
